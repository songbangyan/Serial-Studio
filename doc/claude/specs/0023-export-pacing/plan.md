# Plan 0023 — Scheduled dashboard ticks and CSV interval-snapshot logging

> **Phase 2 of 4 — the HOW.** Grounded in a full read of `FrameBuilder.h/.cpp`,
> `CSV/Export.h/.cpp`, `FrameConsumer.h`, `TimerEvents.h`, `DashboardHandler.cpp`
> (tick/reprocess), `CSVExportHandler.cpp`, and `prelude.js` on 2026-07-20.

## Approach (one paragraph)

`FrameBuilder::dashboardTick()` becomes a scheduler: it validates and seeds exactly as
today, then sets a pending flag that a `Misc::TimerEvents::uiTimeout` slot drains by
running the current synchronous body (`republishFrames(true)`), so any number of calls
per UI window coalesce into one bounded republish. The CSV worker gains an interval
mode: a new `exportInterval` (ms, QSettings-backed, 0 = per-frame default) property on
`CSV::Export`, mirrored to the worker via queued invoke; when non-zero, `processItems`
only forward-fills `m_lastFinalValues` and a worker-thread snapshot timer writes one
full-schema row every N ms (draining the queue first). The BADAQ project's control
script stops calling `dashboardTick()` per frame while the audio source is live, and its
fleet watchdog pins `io.getLatestFrame` to source 0 so audio traffic can no longer mask
a dead CAN bus.

## Architecture & data flow

**Tick scheduling (main thread only).** `DashboardHandler::tick` (GUI thread, via the
control-API marshaller) calls `FrameBuilder::dashboardTick()`, which now: checks
ProjectFile mode + non-empty `m_frame` (as today), seeds `m_sourceFrames` if empty (as
today), sets `m_tickPending = true`, returns true. A new connection in
`setupExternalConnections()` — `TimerEvents::uiTimeout` → lambda — runs
`if (m_tickPending) { m_tickPending = false; (void)dashboardTickNow(); }` where
`dashboardTickNow()` is the current body's tail (`republishFrames(true)`).
`republishFrames` re-validates operation mode at fire time; `onOperationModeChanged` and
the disconnect branch of `onConnectedChanged` also clear `m_tickPending` so a stale tick
never publishes a template frame after teardown. `reprocessFrames()` is untouched.

**CSV interval mode (worker thread).** `CSV::Export` stores the interval in QSettings
(`CSVExportInterval`, ms) and forwards changes to the worker with
`QMetaObject::invokeMethod(..., Qt::QueuedConnection)`, same pattern as
`setTemplateFrame`. The worker owns a `QTimer` created lazily on its own thread
(`Qt::PreciseTimer`), started when interval > 0, stopped at 0. `processItems` keeps file
creation and the forward-fill loop identical; only the row-write block becomes
`if (m_snapshotIntervalMs == 0) writeRow(elapsed)` per item. The snapshot slot does:
`if (!consumerEnabled()) return; drain pending items (processData); if (isResourceOpen()
&& m_sawDataThisSession) { writeRow(now - reference); flush; }`. The drain-first step
bounds cell staleness by the snapshot interval instead of the 1000 ms batch timer at low
frame rates. `closeResources()` resets the session flag; the timer keeps running and
no-ops while the file is closed.

**BADAQ control script.** `latestFrame()` passes `{ sourceId: 0 }` so both the fleet
watchdog and the tick trigger key off the CAN source; the audio stream (48 kHz) can no
longer feed the watchdog or drive the render pass. `dashboardTick()` still fires on every
new CAN frame — it must, because the BADAQ CAN parser writes only data tables and returns
no channels, so the table-driven CAN datasets render and reach the export sinks
*exclusively* through the tick's transform pass. Gating the tick off (the originally
chosen "gate on audio" mitigation) would freeze every CAN temperature/diagnostic dataset
and stop recording it during vibration runs. The audio-firehose CSV problem is solved by
interval mode; the per-CAN-frame tick is bounded by the coalescing in this spec.

## Hotpath & threading impact

- **Touches the hotpath?** Yes — `FrameBuilder` (read in full this session) and the CSV
  consumer. The parse pipeline (`hotpathRxFrame` → span lane → `hotpathTxFrame`) is
  untouched; `dashboardTick` is not on the parse path. The CSV worker's per-frame path
  gains one branch on a worker-local `int`. `--benchmark-hotpath` must be run by the
  maintainer post-merge (AC4); no gate-relevant loop changes.
- **New cross-thread signal/slot?** One new same-thread connection
  (`TimerEvents::uiTimeout` → FrameBuilder lambda, AutoConnection = direct, matching the
  existing `timeout1Hz` wiring). Worker-side: one new queued invoke (interval setter) and
  a worker-owned QTimer — both stay inside the established FrameConsumer pattern; no
  mutexes, SPSC queue untouched.
- **New input to a cached hotpath flag?** No. `m_tickPending` is not read by the parse
  path; `m_anyAsyncSink` inputs unchanged. The snapshot interval is worker-thread state,
  synced only by queued invoke.
- **Timestamp ownership** — unchanged. Snapshot rows are worker-side wall-of-session
  values computed from `steady_clock::now()` against `m_referenceTimestamp` through the
  existing `monotonicFrameNs` guard; frames are never re-stamped.

## Tradeoffs & alternatives considered

| Decision | Options | Chosen + why |
|----------|---------|--------------|
| CSV blowup fix | Interval snapshot mode; per-source files; both | Interval mode (maintainer, 2026-07-20): CSV becomes the test-cell trend log; full-rate belongs to MDF4/Sessions. Per-source files still grow at 48 k rows/s and add a settings surface. |
| Tick export fan-out | Keep `feedExports=true` coalesced; dashboard-only | Keep, coalesced (maintainer): table-driven projects rely on tick rows for export; UI-rate bound makes them safe. |
| Tick return contract | Block until republish (today); schedule + `scheduled` response | Schedule: per-frame callers stop paying a BlockingQueuedConnection round-trip per call; `dashboard.reprocess` keeps the synchronous path. |
| Snapshot cadence source | Reuse batch timer; separate worker QTimer | Separate timer: batch timer is I/O batching (1000 ms, threshold-driven) and must keep draining independently of row cadence. |
| BADAQ script | Gate tick on audio liveness (maintainer's pick); pin watchdog+tick to CAN source, keep ticking | Pin to CAN source: the maintainer's "gate on audio" would freeze/stop-recording the table-driven CAN datasets (CAN parser returns no channels, so only the tick renders them). Pinning to sourceId 0 achieves the intent (decouple from the 48 kHz audio) without the freeze; coalescing + CSV interval mode bound the output. |
| Interval persistence | QSettings; project key | QSettings, matching `CSVExport`; a project key would need schema/writer churn for an app-level concern. |

## Risks & mitigations

- **Scripts depending on synchronous tick read-back** — `dashboard.getData` right after
  `dashboardTick()` may now be one UI window stale. Mitigation: documented in the command
  description + SDK docs; `dashboard.reprocess` stays synchronous.
- **Stale pending tick after project/connection teardown** — cleared in
  `onOperationModeChanged` and the disconnect branch of `onConnectedChanged`;
  `republishFrames` re-checks mode at fire time (defense in depth).
- **Snapshot rows with stale cells at low frame rates** — snapshot slot drains the queue
  before writing (R4); staleness is bounded by the snapshot interval.
- **Silent-breakage classes** (common-mistakes.md): timestamp capture (no re-stamping —
  snapshot elapsed time is explicitly session-relative, not a frame timestamp); queued-vs-
  direct (uiTimeout hop is same-thread direct; all worker mutations queued); ComboBox
  restore race in Setup.qml (guard like neighbors).
- **Benchmark floors** — exporter reference rows exercise the per-frame path; that path
  changes by one branch. Maintainer runs `--benchmark-hotpath` (AC4).
